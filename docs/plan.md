# MapStory 开发计划

## 1. 项目目标梳理
MapStory 的目标是构建一个“历史事件/世界线”管理工具，支持研究与创作场景：
- 以数据库为核心，管理带有时间、地点、人物、事件、优先级、备注的事件条目。
- 提供高效录入、更新、检索与归纳。
- 逐步扩展为可导入导出、可审核、可视化的系统。

## 2. 当前状态（基于 README）
已完成：
- 数据库输入、存储、更新。
- 按时间段/地点范围/人物检索并集中显示。
- 交互式输入输出（CLI 形态）。

待完成：
- 时间排序能力完善。
- 历史纪年与公元纪年转换、年号数据库、年龄计算。
- Excel 导入导出、纯文本导入、维基数据导入。
- 事件优先级审核流。
- 更友好的界面（CLI TUI 或 Web）。

## 3. 开发原则
- 先“数据正确性”，再“自动化能力”，最后“界面体验”。
- 所有扩展功能优先做可回滚的最小版本（MVP）。
- 每个阶段都要有可验证输出（测试、示例数据、使用文档）。

## 3.1 优先级调整（2026-04）
- 提升“界面升级”为高优先级，目标是尽快交付可在网页上操作的版本。
- 下调“外部数据接入”为低优先级，放在核心能力稳定和 Web MVP 之后。

## 4. 分阶段路线图

### 阶段 A：核心能力加固（高优先级）
目标：把已有功能做稳，形成可靠的数据与查询基线。

任务：
1. 统一事件数据模型与字段校验规则。
2. 完成时间排序策略（含空时间、模糊时间、冲突时间处理）。
3. 为 CRUD 与检索增加基础测试。
4. 补齐错误处理与日志（输入错误、数据库异常）。

交付物：
- 稳定的数据模型与排序实现。
- 可运行测试集。
- 更新后的使用说明。

验收标准：
- 新增/修改/查询/排序在示例数据上结果可复现。
- 关键路径测试通过。

### 阶段 B：时间系统扩展（高优先级）
目标：支持历史时间语义，提升历史研究可用性。

任务：
1. 定义时间表示层：公元纪年、历史纪年、不确定时间。
2. 实现历史纪年 <-> 公元纪年的双向转换接口。
3. 设计并接入年号变更数据表（先覆盖常用朝代，后续扩充）。
4. 基于人物生卒年与事件时间实现年龄计算。

交付物：
- 时间转换模块。
- 年号数据库初版。
- 年龄计算工具函数与展示字段。

验收标准：
- 典型朝代案例转换结果正确。
- 年龄计算在边界场景（缺失生日、跨年）下行为可解释。

### 阶段 C：导入导出能力（中优先级）
目标：提高数据流转效率，降低录入成本。

任务：
1. 设计统一导入导出格式（内部 JSON Schema + Excel 映射）。
2. 实现 Excel 导出（年表/月表，支持地域并列视图）。
3. 实现 Excel 导入（字段映射、重复检测、冲突提示）。
4. 实现纯文本导入 MVP（规则模板优先，AI 解析作为可选插件）。

交付物：
- 导入导出模块。
- 示例模板文件（Excel/文本）。
- 冲突与错误报告机制。

验收标准：
- 往返导入导出后数据不丢字段。
- 对非法输入给出可读报错。

### 阶段 D：审核工作流（中优先级）
目标：把“史实/存疑/自设/删减”从手工习惯升级为系统流程。

任务：
1. 明确优先级状态机与状态迁移规则。
2. 增加审核记录（审核人、时间、备注）。
3. 提供批量审核与筛选视图。
4. 在查询结果中突出显示未审核或存疑事件。

交付物：
- 审核状态模型。
- 审核操作接口与查询过滤。

验收标准：
- 状态迁移可追溯。
- 审核后查询结果与统计一致。

### 阶段 E：外部数据接入（低优先级）
目标：减少手工维护成本，扩展数据来源。

任务：
1. 先做离线抓取/导入原型（避免在线依赖阻塞主流程）。
2. 设计维基事件抓取器（频率限制、重试、字段清洗）。
3. 增加来源标记与可信度标注。
4. 建立去重策略（时间+地点+人物+事件近似匹配）。

交付物：
- 抓取与清洗脚本。
- 来源追踪字段。

验收标准：
- 抓取数据可稳定落库。
- 去重误判率在可接受范围。

### 阶段 F：界面升级（高优先级）
目标：提升可用性与演示能力。

任务：
1. 采用 Web（Flask）路线，优先交付浏览器可用版本。
2. 先交付最小界面：新增事件、检索、审核状态编辑、导出。
3. 增加分页、排序、过滤、详情页。
4. CLI 保留为兼容入口，不阻塞 Web MVP 进度。

交付物：
- 可演示 UI 原型。
- 对应接口层改造（服务层与展示层解耦）。

验收标准：
- 用户可在界面内完成核心闭环（录入-检索-审核-导出）。

## 5. 建议里程碑
- M1（2 周）：阶段 A 完成。
- M2（3-4 周）：阶段 B 完成。
- M3（2-3 周）：阶段 F（Web MVP）完成并可演示。
- M4（2-3 周）：阶段 C + D 完成最小可用版本。
- M5（2-3 周）：阶段 E 视资源推进。

## 6. 风险与应对
- 历史时间表达复杂：先明确“可表示范围”，超出范围留注释字段。
- 外部数据质量不稳定：强制来源记录 + 人工审核入口。
- 功能膨胀：每阶段限定 MVP，不在同阶段并行引入高耦合需求。

## 7. 下一步执行清单（可直接开工）
1. 建立任务分支，启动 Flask Web MVP（事件列表、新增、编辑、检索）。
2. 抽离服务层接口，保证 CLI 与 Web 共用同一套业务逻辑。
3. 补充 Web 端最小验收用例（新增-检索-修改状态-导出闭环）。
4. 完成后再并行推进阶段 A/B 的核心能力加固与时间系统扩展。
