# MapStory 开发说明

## 1. 项目定位

MapStory 是一个以“历史事件记录”为核心的数据管理工具。当前仓库已经实现了三层可用能力：

- 核心存储层：基于 SQLite 的事件 CRUD、筛选、排序、软删除。
- CLI 层：支持命令行创建、更新、查询、删除、搜索。
- Web 层：基于 Flask 的本地 Web UI 和 JSON API。

从代码现状看，这个项目已经不是纯概念原型，而是一个“可运行的单机应用 MVP”。它的优势是结构比较直接、依赖少、上手成本低；它的主要问题是“设计目标比当前实现超前”，也就是文档里设想了较多能力，但代码层面仍以一个简单事件库为主。

## 2. 仓库结构总览

当前仓库的核心目录如下：

```text
MapStory/
├── mapstory/
│   ├── __init__.py
│   ├── mapstory.py
│   ├── constants.py
│   ├── errors.py
│   ├── models.py
│   ├── store.py
│   ├── validators.py
│   ├── time_utils.py
│   ├── cli/
│   ├── output/
│   ├── import_/
│   └── ext/
├── tests/
├── docs/
├── data/
├── main.py
├── pyproject.toml
└── README.md
```

可以把它理解成下面这条主链路：

`CLI / Web / 交互模式 -> EventStore -> SQLite`

其中 `EventStore` 是真正的中心，其他入口基本都是对它的封装。

## 3. 核心模块分析

### 3.1 `mapstory/store.py`

这是项目最重要的文件，承担了几乎全部核心业务逻辑：

- 初始化 SQLite 连接。
- 自动建表和补字段迁移。
- 新增、读取、更新、删除事件。
- 按时间、地点、人物、优先级做检索。
- 维护 `time_year/time_month/time_day/time_sort_bucket` 这些排序辅助字段。

这个设计的优点是简单直接，所有入口都复用同一套数据规则；缺点是职责偏重，已经同时承担了：

- 数据访问层
- 一部分业务规则
- 一部分兼容接口
- 一部分迁移逻辑

如果后续继续扩展导入、审核、地图、历史纪年转换等功能，`EventStore` 很容易继续膨胀。

### 3.2 `mapstory/cli/`

CLI 分成两层：

- `main.py` 负责参数解析和命令分发。
- `commands.py` 负责把参数转成 `EventStore` 调用。
- `interactive.py` 提供一个简化的命令行交互模式。

这一层整体比较薄，方向是对的。问题在于：

- 交互模式和 argparse CLI 使用了两套输入逻辑，存在重复。
- CLI 目前主要围绕 `event` 资源，未来若扩展导入、导出、审核、年号库，命令组织可能需要重构。

### 3.3 `mapstory/ext/web.py`

Web 层是一个 Flask MVP，并且实现得比 README 里描述的更完整一些：

- HTML 页面：列表、新建、编辑、详情、删除。
- JSON API：`/api/events` 和 `/api/events/<id>`。
- 页面模板直接以内嵌字符串维护。

它的优点：

- 本地启动简单。
- 与存储层复用充分。
- UI 和 API 语义基本一致。

它的不足：

- HTML/CSS/路由全部在一个文件里，维护成本会随功能增长快速上升。
- 请求级 `EventStore` 管理方式可用，但仍比较原始。
- 模板内嵌在 Python 字符串中，不利于前端迭代、样式复用和测试。

### 3.4 `mapstory/time_utils.py`

时间工具模块目前主要做三件事：

- 解析不完整时间，如 `YYYY`、`YYYY-MM`、`YYYY-MM-DD`。
- 生成排序粒度。
- 做 UTC 和本地显示之间的转换。

这里是一个“目标很大、当前实现较轻”的典型模块。现在的逻辑足够支持基础排序和显示，但距离“历史纪年法转换”“复杂时间表达”“严格时区模型”还有明显距离。

尤其需要注意的一点是：当前 `time_iso` 这个命名带有“完整 ISO 时间戳”的语义，但实际代码里经常存放的是：

- `1911`
- `1911-10`
- `1911-10-10`
- `-221`
- 甚至非严格日期文本

也就是说，字段名和真实数据模型之间存在偏差，这会影响后续 API、导出和前端理解。

### 3.5 `mapstory/models.py`

这里定义了 `Event`、`TimeInfo`、`Location` 三个 dataclass，看起来像领域模型层，但目前项目主体并没有真正围绕这些模型组织，而是更多直接使用 `sqlite3.Row`。

这意味着目前的 `models.py` 更像“未来意图”而非“当前核心实现”。它并没有错，但需要做一个选择：

- 要么继续推进，让 dataclass 真正成为内部统一的数据对象。
- 要么接受当前以 Row/dict 为中心的实现，减少这层抽象，避免名义模型和实际数据流脱节。

### 3.6 `mapstory/output/`、`mapstory/import_/`、`mapstory/ext/` 其他文件

这些模块整体处于“部分实现 + 部分占位”的状态：

- `output/` 已实现 JSON/CSV 文本导出格式化。
- `import_/excel.py` 仍是占位。
- `ext/time_conversion.py`、`ext/era_database.py`、`ext/age_calc.py` 代表未来方向，但不是当前主链路重点。

这说明仓库已经有明确扩展方向，但尚未形成真正稳定的插件化或子系统边界。

## 4. 当前数据模型与执行流程

### 4.1 事件表结构

`events` 表当前大致包含这些字段：

- 基础字段：`id`、`event`、`priority`、`remark`
- 时间字段：`time_iso`、`time_year`、`time_month`、`time_day`、`time_sort_bucket`、`time_note`
- 地点字段：`lat`、`lon`、`location_note`
- 人物字段：`persons`
- 审计字段：`created_at`、`updated_at`、`deleted_at`

这里最关键的设计是：

- 用一个相对宽松的 `time_iso` 保存输入时间文本。
- 再拆出 `time_year/month/day/sort_bucket` 作为检索和排序辅助字段。

这个思路很实用，尤其适合“不完整时间”场景，但需要更明确地写清“字段语义”。

### 4.2 创建事件流程

创建事件时的主要流程如下：

1. CLI 或 Web 收集输入。
2. `validators.py` 做文本、优先级、坐标的规范化和校验。
3. `time_utils.parse_time_components()` 解析时间文本，生成排序辅助字段。
4. `EventStore.create_event()` 写入 SQLite。
5. 返回事件记录给 CLI 表格输出或 Web/API 响应。

### 4.3 查询与排序流程

项目当前最重要的查询能力来自 `EventStore.search_events()`：

- 时间范围：基于 `time_year`
- 地点范围：基于 `lat/lon`
- 人物模糊查询：基于 `persons LIKE`
- 事件模糊查询：基于 `event LIKE`
- 地点备注查询：基于 `location_note LIKE`
- 优先级精确匹配

排序上主要支持两种：

- `time`：按 `time_sort_bucket -> year -> month -> day`
- `created`：按创建时间倒序

这套设计已经足够支撑“小型个人史料库/世界线库”的常见操作。

## 5. 现状优点

这个仓库目前最值得肯定的地方有这些：

- 依赖极少，部署门槛低，`SQLite + Flask` 很适合个人工具。
- 主干清晰，`EventStore` 作为单一事实来源，CLI 与 Web 共用逻辑。
- 有基础测试，尤其覆盖了时间排序、Web CRUD 和古代负年份场景。
- 对软删除、基础迁移、输入校验已经有初步考虑，不是纯 demo。
- 中文语义明确，比较贴近实际使用场景，而不是抽象过度。

## 6. 当前主要问题

### 6.1 文档设想与代码实现存在落差

`docs/design.md` 里描述的是一个更完整的中长期架构，但代码当前还处在较早阶段。典型差异包括：

- 文档强调 dataclass 模型，但主流程实际使用 `sqlite3.Row`。
- 文档强调 UTC 输入输出策略，但当前 `time_iso` 并不是严格 UTC 时间戳。
- 文档列出的很多模块和测试结构还未落地。

这不会立刻导致程序不可用，但会让新开发者误判系统成熟度。

### 6.2 时间字段语义不够统一

`time_iso` 这个名字容易让人误解为“标准 ISO 时间戳”，但它目前更像“宽松日期文本”。这会带来几个隐患：

- API 使用方会误以为它总能被 `datetime.fromisoformat()` 正确解析。
- 展示层和存储层的规则容易越写越散。
- 后续做导入、导出、前端筛选时，边界行为会越来越多。

### 6.3 `EventStore` 过于集中

当前它既是 repository，又夹带业务规则和兼容接口。短期开发很快，长期会出现这些问题：

- 单文件体积继续增长。
- 测试粒度不清晰。
- Web/CLI 若增加更多业务，会继续把逻辑压进 store。

### 6.4 Web 层单文件过大

`ext/web.py` 里同时放了：

- 样式
- 模板
- 路由
- 表单解析
- API 组装

对 MVP 来说没问题，但已经接近应该拆分的时机。

### 6.5 测试覆盖面仍然有限

现有测试覆盖了主链路，但还缺少这些更容易出问题的场景：

- 更新时清空字段的行为
- 软删除后的列表/详情/API 边界
- 非法时间文本的展示与排序行为
- CLI 参数级测试
- 导出模块测试
- 数据库迁移兼容测试

### 6.6 占位模块较多，边界还不清楚

导入、年号转换、年龄计算等方向都已经露头，但还没有统一扩展策略。这样继续迭代时，容易出现：

- 名字上属于“扩展”，但实际上深度耦合核心数据结构
- 每个子模块各自实现输入/输出规则
- 最终形成多个半成品子系统

## 7. 改进建议

下面按优先级给出建议。

### 7.1 第一优先级：统一数据语义和文档

这是最值得先做的一步。

建议：

- 明确 `time_iso` 的真实语义。
- 二选一：
  - 方案 A：把它改名为 `time_expr` / `time_raw`，承认它是宽松时间表达。
  - 方案 B：强制它只保存标准 ISO/日期格式，另设字段保存原始历史纪年文本。
- 更新 `README.md`、`docs/design.md`、API 返回说明，让文档和代码一致。

原因：

- 这会直接降低后续开发歧义。
- 也能为导入、筛选、前端展示建立稳定契约。

### 7.2 第二优先级：把 `EventStore` 做轻量拆层

不需要一次性上复杂架构，但建议至少做以下拆分：

- `store.py` 只保留数据库访问和 SQL。
- 新建 `services/events.py` 负责创建、更新、搜索时的业务拼装。
- 新建 `schemas.py` 或 `serializers.py` 负责 Row/dict/API payload 转换。

这样可以保留当前简单实现风格，同时让职责更清晰。

### 7.3 第三优先级：拆分 Web 层

建议把 `ext/web.py` 逐步拆成：

- `ext/web/app.py`：应用工厂
- `ext/web/routes.py`：页面和 API 路由
- `ext/web/forms.py`：参数解析和校验
- `ext/web/templates/`：Jinja 模板

这一步的收益很直接：

- 模板能独立维护
- 路由可测试性更高
- 后续若增加分页、筛选项、地图视图，不会继续堆到一个文件里

### 7.4 第四优先级：决定 dataclass 模型是否要真正启用

这里不要模糊，建议明确路线：

- 如果想让项目更“领域化”，就让 `Event` 成为真正的内部对象，store 层负责 Row <-> Event 转换。
- 如果更偏向实用型脚本应用，就弱化 `models.py` 的存在感，减少无效抽象。

我更倾向第一种，但前提是团队愿意接受一轮结构整理。

### 7.5 第五优先级：补测试，而不是先补功能

比较建议优先增加这些测试：

- `EventStore.update_event()` 对空值和部分更新的行为
- `delete_event()` 软删除/硬删除差异
- `search_events()` 的组合筛选
- CLI 命令行参数解析与错误输出
- 时间显示在负年份、模糊月份、非法文本时的表现
- schema migration 场景

原因很简单：这个项目现在已经进入“加功能前先稳住骨架”的阶段。

### 7.6 第六优先级：为导入/扩展能力设边界

建议不要直接在 `ext/` 里持续堆新文件，而是先回答两个问题：

1. 哪些能力是“核心事件系统”的一部分？
2. 哪些能力是“外部数据接入”或“辅助计算”？

一个更清晰的方向可能是：

- `mapstory/core/`：事件、时间、校验、查询
- `mapstory/interfaces/`：CLI、Web、API
- `mapstory/importers/`：Excel、文本、Wikipedia
- `mapstory/extensions/`：年号库、年龄计算、地理编码

不一定现在就改目录，但后续新增功能时最好遵守类似分层。

## 8. 对后续版本的建议路线

如果按“投入小、收益高”的顺序推进，我建议是：

1. 先统一时间字段语义，并同步文档。
2. 拆分 Web 文件和 Row/payload 转换逻辑。
3. 增补测试，尤其是更新、删除、搜索、迁移。
4. 再开始做导入、年号数据库、地图或审核流。

换句话说，当前仓库最需要的不是“继续铺更多功能点”，而是“把现有 MVP 打磨成稳定内核”。

## 9. 结论

MapStory 当前最核心的特点是：

- 已经具备可用的 SQLite 事件库骨架。
- CLI 和 Web 双入口都能工作。
- 代码整体可读，适合继续演进。

但它也处于一个很典型的拐点：

- 如果继续按现在的方式加功能，代码会越来越依赖“单文件 + 约定”。
- 如果现在做一次温和整理，这个项目就有机会从“AI 辅助搭起来的 MVP”走向“可长期维护的小型应用”。

对这个仓库最务实的判断是：基础已经有了，下一阶段应该优先做结构收敛，而不是功能发散。
