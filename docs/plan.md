# MapStory 开发计划

## 1. 项目目标梳理
MapStory 的目标是构建一个“历史事件/世界线”管理工具，支持研究与创作场景：
- 以数据库为核心，管理带有时间、地点、人物、事件、优先级、备注的事件条目。
- 提供高效录入、更新、检索与归纳。
- 逐步扩展为可导入导出、可审核、可视化的系统。

## 2. 当前状态（基于 README）
已完成：
- 数据库输入、存储、更新。
- 按时间段/地点范围/人物检索并集中显示。
- 交互式输入输出（CLI 形态）。

待完成：
- 时间排序能力完善。
- 历史纪年与公元纪年转换、年号数据库、年龄计算。
- Excel 导入导出、纯文本导入、维基数据导入。
- 事件优先级审核流。
- 更友好的界面（CLI TUI 或 Web）。

## 3. 开发原则
- 先“数据正确性”，再“自动化能力”，最后“界面体验”。
- 所有扩展功能优先做可回滚的最小版本（MVP）。
- 每个阶段都要有可验证输出（测试、示例数据、使用文档）。

## 3.1 执行顺序调整（2026-04）
- 按以下顺序推进：1. API 重构 -> 2. 模块拆分迁移 -> 3. Web UI。
- 不要求保留旧 CLI 兼容层，允许直接采用新命令模型。
- 目标是本地自托管、个人使用优先，先交付可在电脑和手机浏览器操作的版本。

## 4. 三阶段执行主线

### 阶段 1：API 重构（最高优先级）
目标：定义清晰、一致、可扩展的 CLI 与 Web 统一语义。

任务：
1. 以 Event 作为唯一核心资源，冻结字段定义和校验规则。
2. 将 CLI 改为资源化命令模型：event create/list/search/get/update/delete。
3. 统一输出约定：table 与 json 两种格式。
4. 定义错误码与错误消息结构，为 Web API 复用。

交付物：
- API 规范文档（CLI + Web 对齐）。
- 新 CLI 命令骨架与参数解析实现。
- 新命令回归测试。

验收标准：
- 新命令可完成完整 CRUD。
- 所有校验与错误消息可复现。
- 测试覆盖新增/查询/更新/删除/获取单条。

### 阶段 2：模块拆分迁移（高优先级）
目标：把单文件逻辑拆分为小模块，降低耦合并便于后续 Web 接入。

任务：
1. 拆分模块：constants、errors、validators、time_utils、store、cli、interactive。
2. 先迁移纯函数（校验/时间解析），再迁移存储层，最后迁移 CLI。
3. 保持数据库结构和核心行为一致，避免迁移期数据风险。
4. 完成迁移后清理废弃代码与重复实现。

交付物：
- 新包结构与模块职责说明。
- 迁移后的可运行 CLI。
- 迁移回归测试报告。

验收标准：
- 现有测试全绿。
- 代码职责边界清晰，单文件不再过大。
- 数据库读写行为与预期一致。

### 阶段 3：Web UI（高优先级）
目标：交付本地自托管可用的图形化界面，支持电脑与手机浏览器访问。

任务：
1. 使用 Flask 实现单体 Web 应用（页面 + API）。
2. 先交付 MVP 页面：事件新增、列表、检索、编辑、删除。
3. 增加移动端可用布局与基础交互反馈。
4. 增加本地访问模式说明：127.0.0.1 与局域网访问方式。

当前进展：
- 已实现 Flask Web MVP。
- 页面包括列表、详情、编辑、新增、删除。
- API 包括 `GET/POST /api/events` 和 `GET/PATCH/DELETE /api/events/<id>`。

交付物：
- Web MVP（本地可运行）。
- 页面与 API 联调通过。
- 本地启动与手机访问说明文档。

验收标准：
- 用户可在 UI 完成完整 CRUD。
- 桌面与手机浏览器都可完成关键流程。
- 本地数据落库稳定。

## 5. 里程碑（按执行顺序）
- M1（1-2 周）：阶段 1 API 重构完成。
- M2（1-2 周）：阶段 2 模块拆分迁移完成。
- M3（2-3 周）：阶段 3 Web UI MVP 完成并可本地跨设备访问。

## 6. 风险与应对
- API 重写期变动大：先冻结字段和错误语义，再动实现。
- 拆分迁移易引入回归：每完成一个子模块即跑测试。
- 手机访问受网络限制：优先同一局域网，必要时再评估内网穿透。
